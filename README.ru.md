# opencode-minispec

[English version](README.md)

> Проект активно развивается. Команды, форматы документов и детали процесса могут измениться.

`opencode-minispec` — это плагин OpenCode для документно-ориентированного процесса разработки ПО, рассчитанного на множество сессий.

Вместо того чтобы хранить состояние проекта в окне контекста или в отдельном документе режима планирования, minispec хранит его в файлах: концепт, требования, технические детали, бэклог, завершенные задачи, спецификации и уроки.  
Каждая задача сначала описывается письменно, затем реализуется по этой спецификации, а после принятия переносится обратно в живые документы.  
Следующая сессия начинается с файлов, а не с памяти модели.

## Установка

Добавьте пакет плагина в конфиг OpenCode:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": ["opencode-minispec"]
}
```

После обновления `opencode.json` перезапустите OpenCode.

Пользовательские команды OpenCode живут в плоском пространстве имен, поэтому minispec добавляет префикс `mspc-` к каждой команде, чтобы избежать конфликтов со встроенными командами вроде `/init`.

## Процесс

minispec работает итеративными циклами:

```text
/mspc-init              Инициализация проекта
    |
/mspc-explore           Исследование (в любой момент)
    |
    v
/mspc-task-new  ->  /mspc-task-impl  ->  /mspc-task-accept
      (спецификация)    (реализация)         (перенос и архив)
    ^                                       |
    └───────────────────────────────────────┘
                    повтор

/mspc-task-quick       Тривиальное изменение без спецификации
/mspc-review           Аудит состояния — может добавить задачи в TODO
```

**Фаза 1 — Инициализация.** Запустите `/mspc-init` с описанием идеи или файлом идеи. Команда создает структуру документов minispec, инструкции проекта, базовые документы и первую версию `minispec/TODO.md` на основе идеи.

**Фаза 2 — Исследование.** Используйте `/mspc-explore` в любой момент, чтобы разбирать идеи, открытые вопросы, компромиссы или новые направления. Команда сохраняет итог в `minispec/explorations/` и обновляет документы проекта, когда исследование меняет их содержимое. На практике это основной способ добавлять или уточнять элементы бэклога после инициализации.

`minispec/TODO.md` — это рабочий бэклог. Он может пополняться четырьмя способами:
- `/mspc-init` создает начальные элементы
- `/mspc-explore` добавляет задачи, вопросы или исследовательские элементы, когда появляется конкретная работа
- `/mspc-review` может переносить результаты проверки в `minispec/TODO.md`
- `/mspc-task-new <description>` может создать новую запись задачи перед написанием спецификации

Поскольку бэклог хранится в файле, вы также можете редактировать `minispec/TODO.md` напрямую, если так проще.

**Фаза 3 — Разработка.** Когда элемент бэклога готов к работе, выберите его идентификатор из `minispec/TODO.md` и выполните:
1. `/mspc-task-new T0001` — создает подробную спецификацию в `minispec/specs/`
2. `/mspc-task-impl T0001` — реализует спецификацию, запускает тесты и проверяет критерии приемки
3. `/mspc-task-accept T0001` — переносит содержимое спецификации в `minispec/REQS.md`, `minispec/TECH.md`, `minispec/TESTS.md`, архивирует задачу и удаляет файл спецификации

**Фаза 4 — Аудит.** Периодически запускайте `/mspc-review`, чтобы проверять документы, код, тесты и активные спецификации на несоответствия.

### Структура проекта, которую создает minispec

minispec владеет сгенерированным каталогом `minispec/` и корневым `AGENTS.md`. Исходный код, тесты и все остальные файлы проекта принадлежат пользователю и могут использовать любую подходящую структуру.

```text
AGENTS.md                   Инструкции для OpenCode
minispec/
  CONCEPT.md                Видение и северная звезда проекта
  REQS.md                   Только реализованные требования
  TECH.md                   Только реальная техническая архитектура
  TESTS.md                  Реестр тестов (автоматических и ручных)
  TODO.md                   Бэклог с идентификаторами задач (T0001, T0002, ...)
  DONE.md                   Архив завершенных задач
  LESSONS.md                Неочевидные выводы: развороты, отмененные подходы, подводные камни
  specs/                    Временные спецификации по задачам
  explorations/             Артефакты исследований и мозговых штурмов
```

## Команды

### `/mspc-init`

Инициализирует новый проект из описания идеи.

```text
/mspc-init Хочу CLI-инструмент, который преобразует CSV в JSON и поддерживает потоковую обработку
/mspc-init path/to/idea-file.md
```

Эта команда создает структуру minispec выше и заполняет:
- `minispec/CONCEPT.md` — выжимку видения, мотивации, основной идеи, ключевых решений, планируемых функций и открытых вопросов
- `minispec/TODO.md` — начальный бэклог с разделами `Development`, `Open Questions` и `Exploration`
- `minispec/REQS.md`, `minispec/TESTS.md`, `minispec/DONE.md`, `minispec/LESSONS.md` — каркасы живых документов
- `minispec/TECH.md` — каркас, описывающий только реальное состояние, который позже пополняется через `/mspc-task-accept`
- `AGENTS.md` — снимок проекта, принципы, философию документов, описание рабочего процесса и правила для обычных диалогов OpenCode

### `/mspc-explore`

Исследует, анализирует и обсуждает идеи на любом этапе проекта.

```text
/mspc-explore [topic or question]
```

Результат сессии:
- итоговый файл в `minispec/explorations/<date>-<topic>.md`
- обновления `minispec/CONCEPT.md`, если изменилось видение
- новые элементы в `minispec/TODO.md`, если появилась конкретная работа

Вне `/mspc-explore` сгенерированный `AGENTS.md` все равно поощряет структурированные обсуждения: вопросы блоками, короткую проверку формулировки задачи, явные предположения и компромиссы, а также краткие итоги. Но только `/mspc-explore` остается явным способом сохранять долговечные артефакты и обновлять документы.

### `/mspc-task-new`

Создает задачу с подробной спецификацией.

```text
/mspc-task-new T0001
/mspc-task-new T0001 добавить авторизацию через OAuth
/mspc-task-new добавить поддержку темной темы
```

Команда принимает идентификатор задачи из `minispec/TODO.md` или новое описание. Дальше она выбирает путь по типу задачи:
- **Задача разработки** — создает спецификацию в `minispec/specs/SPEC-T0001.md` с четырехзначными идентификаторами задач (`T0001`, `T0002`, ...)
- **Открытый вопрос / Исследование** — перенаправляет вас в `/mspc-explore`

Если вы передаете новое описание вместо существующего идентификатора, minispec проверяет дубликаты, присваивает следующий идентификатор задачи, подтверждает нужный раздел `minispec/TODO.md` и добавляет запись перед продолжением.

Если работа действительно тривиальна, используйте `/mspc-task-quick`.

### `/mspc-task-impl`

Реализует задачу по ее спецификации.

```text
/mspc-task-impl T0001
```

Команда читает спецификацию и выполняет ее:
1. Планирует реализацию: какие файлы создать или изменить, в каком порядке работать, какие тесты нужны и какие есть неясности
2. Пишет код строго по спецификации
3. Пишет тесты рядом с кодом
4. Проверяет каждый критерий приемки: запускает тесты, проверяет сборку, выполняет lint- и format-шаги и перечисляет ручные проверки, если они нужны
5. Возвращает результаты проверки и перечисляет все ручные проверки, которые еще нужно выполнить

Если для задачи нет спецификации, команда предложит сначала запустить `/mspc-task-new`.

### `/mspc-task-accept`

Принимает завершенную задачу и переносит ее спецификацию в живые документы.

```text
/mspc-task-accept T0001
```

Запускайте эту команду после того, как сами проверили реализацию. Команда:
1. Переносит требования из спецификации в `minispec/REQS.md`
2. Переносит технические детали в `minispec/TECH.md`
3. Переносит тест-кейсы в `minispec/TESTS.md`
4. Архивирует задачу в `minispec/DONE.md`
5. Удаляет задачу из `minispec/TODO.md`
6. Обновляет `minispec/CONCEPT.md`, если реализация изменила видение
7. Фиксирует долговечный вывод в `minispec/LESSONS.md`, если он действительно появился
8. Удаляет файл спецификации

### `/mspc-task-quick`

Быстрый путь для тривиальных изменений без файла спецификации.

```text
/mspc-task-quick T0014
/mspc-task-quick исправить опечатку в README
```

Используйте для изменений, которые слишком малы для полного цикла `/mspc-task-new -> /mspc-task-impl -> /mspc-task-accept`: однострочных исправлений ошибок, мелких правок текста или изолированных переименований.  
Если изменение затрагивает несколько компонентов, имеет неочевидные критерии приемки или требует отдельного архитектурного решения, команда откажется и перенаправит в `/mspc-task-new`.

### `/mspc-review`

Проводит аудит проекта на предмет несоответствий, пробелов и улучшений.

```text
/mspc-review
```

Команда читает все документы, код, тесты и активные файлы спецификаций, затем сопоставляет их между собой:
- **Документы между собой** — соответствуют ли требования техническим компонентам? Согласованы ли `minispec/CONCEPT.md` и `minispec/TODO.md`? Есть ли дубликаты или противоречия?
- **Документы и код** — реализованы ли требования? Соответствует ли `minispec/TECH.md` реальной кодовой базе? Есть ли недокументированное поведение?
- **Качество кода** — проблемы безопасности, запахи кода, обработка ошибок, расхождения в именовании
- **Покрытие тестами** — покрыто ли документированное поведение тестами или понятными ручными проверками? Покрыты ли краевые случаи из активных спецификаций? Актуальны ли ручные тесты?

Находки выдаются по уровню важности с плоскими идентификаторами `F<N>`. После отчета команда может превратить выбранные находки в новые задачи в `minispec/TODO.md`.
